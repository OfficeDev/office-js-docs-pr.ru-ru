## <a name="contribute-using-github"></a>Внесение изменений с помощью GitHub

С помощью GitHub вы можете улучшать эту документацию, не клонируя репозиторий на компьютере. Это самый простой способ создать запрос на включение внесенных изменений в этот репозиторий. Используйте его для внесения небольших изменений, не затрагивающих код. 

**Примечание**. С помощью этого способа невозможно вносить изменения сразу в несколько статей.

### <a name="to-contribute-using-github"></a>Как вносить изменения с помощью GitHub

1. Найдите нужную статью на сайте GitHub.
2. Открыв статью на сайте GitHub, выполните вход (вы можете бесплатно получить учетную запись, [присоединившись к GitHub](https://github.com/join)).
3. Нажмите **значок карандаша** (редактирование файла в ответвлении этого проекта) и внесите свои изменения на вкладке **<>Edit file** (Редактирование файла). 
4. Прокрутите страницу вниз и введите описание.
5. Последовательно нажмите кнопки **Propose file change** (Предложить изменение файла) > **Create pull request** (Создать запрос на включение внесенных изменений).

Теперь вы успешно отправили запрос на включение внесенных изменений. Как правило, запросы на включение внесенных изменений проверяются в течение 10 рабочих дней. 


## <a name="contribute-using-git"></a>Внесение изменений с помощью Git

С помощью Git вы можете вносить существенные изменения, например:

* добавлять код;
* публиковать изменения, влияющие на смысл;
* вносить значительные изменения в текст;
* добавлять новые статьи.

### <a name="to-contribute-using-git"></a>Как внести изменения с помощью Git

1. Если у вас нет учетной записи GitHub, создайте ее на сайте [GitHub](https://github.com/join). 
2. Создав учетную запись, установите Git на компьютере. Выполните действия, описанные в [руководстве по установке Git].
3. Чтобы отправить запрос на включение внесенных изменений с помощью Git, выполните действия, указанные в разделе [Использование GitHub, Git и этого репозитория](#use-github-git-and-this-repository).
4. Вам будет предложено подписать лицензионное соглашение для участников, если вы:

    * являетесь членом группы Microsoft Open Technologies;
    * являетесь участником (вносите свой вклад в документацию), но не работаете в корпорации Майкрософт.

Как члену сообщества, вам нужно подписать лицензионное соглашение для участников (CLA), чтобы вносить значительные изменения в проект. Заполнить и отправить документ достаточно один раз. Внимательно просмотрите документ. Может потребоваться подпись вашего работодателя.

Подписывая лицензионное соглашение для участников (CLA), вы не получаете прав на внесение изменений в основной репозиторий, но члены группы разработчиков Office и группы публикации контента для разработчиков Office смогут просматривать и утверждать ваши изменения. Ваш вклад будет оценен по достоинству.

Как правило, запросы на включение внесенных изменений проверяются в течение 10 рабочих дней.

## <a name="use-github-git-and-this-repository"></a>Использование GitHub, Git и этого репозитория

**Примечание.** Большую часть сведений из этого раздела можно найти в [справочных статьях GitHub].  Если вы знакомы с Git и GitHub, переходите к разделу **Добавление и редактирование содержимого**, в котором подробно описано, как работать с кодом и содержимым в этом репозитории.

### <a name="to-set-up-your-fork-of-the-repository"></a>Настройка ответвления в репозитории

1.  Чтобы вносить свой вклад в этот проект, настройте учетную запись GitHub. Если вы еще не сделали это, перейдите на сайт [GitHub](https://github.com/join) и создайте ее.
2.  Установите Git на своем компьютере. Выполните действия, описанные в [руководстве по установке Git].
3.  Создайте свое ответвление этого репозитория. Для этого нажмите кнопку **Вилка** в верхней части страницы.
4.  Скопируйте свое ответвление на компьютер. Для этого откройте Git Bash. Введите в командной строке следующую команду:

        git clone https://github.com/<your user name>/<repo name>.git

    Затем создайте ссылку на корневой репозиторий с помощью следующих команд:

        cd <repo name>
        git remote add upstream https://github.com/OfficeDev/<repo name>.git
        git fetch upstream

Поздравляем! Вы настроили свой репозиторий. Вам не потребуется повторять эти действия.

### <a name="contribute-and-edit-content"></a>Добавление и редактирование содержимого

Чтобы добавлять и редактировать содержимое было удобнее, выполните указанные ниже действия.

#### <a name="to-contribute-and-edit-content"></a>Как добавлять и редактировать содержимое

1. Создайте новую ветвь.
2. Добавьте или отредактируйте содержимое.
3. Отправьте запрос на включение внесенных изменений в основной репозиторий.
4. Удалите ветвь.

**Важно!** Постарайтесь, чтобы каждая ветвь была связана с одним понятием или одной статьей. Это упрощает работу и снижает риск возникновения конфликтов при объединении. Для новой ветви подходит содержимое следующих типов:

* новая статья;
* исправления орфографии и грамматики;
* применение одного изменения форматирования для большого количества статей (например, применение нового нижнего колонтитула для авторских прав).

#### <a name="to-create-a-new-branch"></a>Создание ветви

1.  Откройте Git Bash.
2.  В командной строке Git Bash введите команду `git pull upstream master:<new branch name>`. При этом создается локальная ветвь, которая копируется из последней главной ветви OfficeDev.
3.  В командной строке Git Bash введите команду `git push origin <new branch name>`. Так вы сообщите GitHub о новой ветви. Новая ветвь появится в вашем ответвлении репозитория на сайте GitHub.
4.  Введите в командной строке Git Bash команду `git checkout <new branch name>`, чтобы переключиться на новую ветвь.

#### <a name="add-new-content-or-edit-existing-content"></a>Добавление или редактирование содержимого

Вы можете перейти к репозиторию на компьютере с помощью проводника. Файлы репозитория находятся в папке `C:\Users\<yourusername>\<repo name>`.

Чтобы отредактировать файлы, откройте их в любом редакторе и измените их. Чтобы создать файл, используйте любой редактор и сохраните новый файл в подходящем расположении в локальной копии репозитория. Периодически сохраняйте файлы во время работы.

Файлы в папке `C:\Users\<yourusername>\<repo name>` — это рабочая копия новой ветви, которую вы создали в локальном репозитории. Изменение файлов в этой папке не влияет на локальный репозиторий, пока вы не сохраните изменение. Чтобы сохранить изменение в локальном репозитории, введите следующие команды в GitBash:

    git add .
    git commit -v -a -m "<Describe the changes made in this commit>"

Команда `add` добавляет изменения в область промежуточного хранения для подготовки к их применению в репозитории. Точка после команды `add` указывает, что вы хотите поместить на промежуточное хранение все добавленные и измененные файлы, рекурсивно проверяя вложенные папки. (Если вы не хотите применять все изменения, вы можете добавить определенные файлы. Вы также можете отменить изменения. Для получения справки введите команду `git add -help` или `git status`.)

Команда `commit` применяет промежуточные изменения в репозитории. Параметр `-m` указывает, что вы предоставляете в командной строке комментарий к изменению. Параметры -v и -a указывать необязательно. Параметр -v обеспечивает предоставление подробных выходных данных для команды, а параметр -a выполняет те же действия, что и уже запущенная нами команда add.

Вы можете сохранять изменения несколько раз во время работы или только один раз, когда все будет готово.

#### <a name="submit-a-pull-request-to-the-main-repository"></a>Отправка запроса на включение внесенных изменений в основной репозиторий

Кода вы завершите работу и будете готовы объединить вашу ветвь с главным репозиторием, выполните указанные ниже действия.

#### <a name="to-submit-a-pull-request-to-the-main-repository"></a>Как отправить запрос на включение внесенных изменений в основной репозиторий

1.  В командной строке Git Bash введите команду `git push origin <new branch name>`. В локальном репозитории параметр `origin` означает репозиторий GitHub, копией которого является ваш локальный репозиторий. Эта команда передает текущее состояние новой ветви, в том числе все изменения, сохраненные на предыдущих шагах, в вашем ответвлении GitHub.
2.  На сайте GitHub перейдите к новой ветви в своем ответвлении.
3.  Нажмите кнопку **Pull Request** (Запрос на включение внесенных изменений) в верхней части страницы.
4.  Убедитесь, что в качестве базовой ветви указана ветвь `OfficeDev/<repo name>@master`, а в качестве главной — `<your username>/<repo name>@<branch name>`.
5.  Нажмите кнопку **Update Commit Range** (Обновить содержимое).
6.  Добавьте название запроса и опишите изменения, которые вы вносите.
7.  Отправьте запрос на включение внесенных изменений.

Его обработает один из администраторов сайта. Ваш запрос появится в разделе "Issues" (Проблемы) на сайте OfficeDev/<repo name>. Когда запрос будет принят, проблема будет устранена.

#### <a name="create-a-new-branch-after-merge"></a>Создание ветви после объединения

После успешного объединения (т. е. принятия запроса на включение внесенных изменений) не продолжайте работать в локальной ветви. Если вы отправите еще один запрос, это может привести к конфликтам при объединении. Чтобы внести еще одно изменение, создайте новую локальную ветвь на основе успешно добавленной ветви и удалите исходную локальную ветвь.

Допустим, локальная ветвь X была успешно объединена с главной ветвью OfficeDev/microsoft-graph-docs, а вы хотите внести дополнительные изменения в объединенное содержимое. Создайте локальную ветвь X2 на основе главной ветви OfficeDev/microsoft-graph-docs. Для этого откройте GitBash и выполните следующие команды:

    cd microsoft-graph-docs
    git pull upstream master:X2
    git push origin X2

Теперь у вас есть локальные копии работы, отправленной в ветви X (они находятся в новой локальной ветви). Кроме того, ветвь X2 содержит все изменения, внесенные другими авторами, так что если в вашей работе используются чужие материалы (например, общие изображения), они будут доступны в новой ветви. Открыв новую ветвь, вы можете убедиться, что ваши предыдущие правки (и чужие материалы) включены в нее.

    git checkout X2

...и проверив содержимое. (Команда `checkout` обновляет все файлы в папке `C:\Users\<yourusername>\microsoft-graph-docs` до текущего состояния ветви X2.) Получив новую ветвь для изменения, вы можете редактировать содержимое и применять изменения обычным образом. Однако во избежание работы с объединенной ветвью (X) рекомендуем удалить ее (см. раздел **Удаление ветви** ниже).

#### <a name="delete-a-branch"></a>Удаление ветви

Успешно объединив свои изменения с главным репозиторием, удалите соответствующую ветвь, так как она больше не требуется.  Всю дальнейшую работу следует выполнять в новой ветви.  

#### <a name="to-delete-a-branch"></a>Как удалить ветвь

1.  В командной строке Git Bash введите команду `git checkout master`. Это гарантирует, что вы не работаете в удаляемой ветви (это не допускается).
2.  Затем введите в командной строке команду `git branch -d <branch name>`. При этом ветвь будет удалена только на вашем компьютере, если она была успешно объединена с репозиторием верхнего уровня. (Вы можете изменить это поведение с помощью флага `–D`, но для начала убедитесь, что это необходимо.)
3.  Наконец, введите в командной строке команду `git push origin :<branch name>` (с пробелом перед двоеточием и без пробела после него).  При этом ветвь будет удалена из вашего ответвления GitHub.  

Поздравляем! Вы успешно внесли свой вклад в проект.

## <a name="faq"></a>Вопросы и ответы

### <a name="how-do-i-get-a-github-account"></a>Как создать учетную запись GitHub?

Заполните форму на странице [Join GitHub](https://github.com/join), чтобы создать бесплатную учетную запись GitHub. 

### <a name="where-do-i-get-a-contributors-license-agreement"></a>Где можно найти лицензионное соглашение с участником? 

Вам будет автоматически отправлено уведомление, если для включения внесенных вами изменений вам нужно подписать лицензионное соглашение с участником (CLA). 

**Вам нужно подписать лицензионное соглашение с участником (CLA), прежде чем вы сможете вносить большие изменения в этот проект**. Вам нужно заполнить и отправить документ всего один раз. Внимательно просмотрите документ. Может потребоваться подпись вашего работодателя.

### <a name="what-happens-with-my-contributions"></a>Что происходит со внесенными мной изменениями?

Когда вы вносите изменения с помощью запроса на включение внесенных изменений, наша команда получает уведомления и проверяет запрос. Вы будете получать уведомления о запросе на включение внесенных изменений от GitHub. Кроме того, вы можете получать уведомления от членов нашей команды, если нам потребуются дополнительные сведения. Если ваш запрос будет одобрен, мы обновим документацию. Мы оставляем за собой право редактировать ваши изменения, исправляя правовые и стилистические ошибки, а также внося ясность.

### <a name="can-i-become-an-approver-for-this-repositorys-github-pull-requests"></a>Могу ли я стать утверждающим для запросов на включение внесенных изменений в репозитории GitHub?

В настоящее время мы не разрешаем внешним участникам утверждать запросы на включение внесенных изменений в этом репозитории.

### <a name="how-soon-will-i-get-a-response-about-my-change-request"></a>Насколько быстро я получу ответ на свой запрос на включение изменений?

Как правило, запросы на включение внесенных изменений проверяются в течение 10 рабочих дней.
