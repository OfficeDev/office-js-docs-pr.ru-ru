## <a name="ways-to-contribute"></a>Как внести свой вклад

Вот несколько способов внести свой вклад в эту документацию:

* Чтобы внести небольшие изменения в статью, [внося вклад с помощью GitHub](#contribute-using-github).
* Чтобы внести большие изменения или изменения, которые связаны с кодом, [внести свой вклад с помощью Git](#contribute-using-git).
* Сообщайте об ошибках документации, переехав в  раздел **Отзыв** в нижней части затронутой статьи, а затем выберите эту страницу для создания проблемы GitHub. Если это не доступно, создайте новую проблему непосредственно на [GitHub.](https://github.com/OfficeDev/office-js-docs-pr/issues)
* Запрос новой документации с [помощью проблем GitHub.](https://github.com/OfficeDev/office-js-docs-pr/issues)

## <a name="contribute-using-github"></a>Внесение изменений с помощью GitHub

Чтобы внести свой вклад в эту документацию, используйте GitHub, не клонировать репо на рабочем столе. Это самый простой способ создания запроса на тягу в этом репозитории. Используйте этот метод, чтобы внести незначительные изменения, которые не связаны с изменениями кода.

**Примечание.** Использование этого метода позволяет одновременно вносить вклад в одну статью.

### <a name="to-contribute-using-github"></a>Участие с помощью GitHub

1. Найдите статью, в которая вы хотите внести свой вклад в GitHub.
2. Во время статьи в GitHub вопишитесь в GitHub (получите бесплатную учетную запись [Join GitHub).](https://github.com/join)
3. Выберите **значок карандаша** (отредактировать файл в вилке этого проекта) и внести изменения в окно файла **<>изменить.**
4. Прокрутите вниз и введите описание.
5. Выберите **Предложение изменения файла Создание** > **запроса на вытягивать**.

Теперь вы успешно отправили запрос на вытягивать. Запросы на вытяги, как правило, рассматриваются в течение 10 бизнес-дней.


## <a name="contribute-using-git"></a>Внесение изменений с помощью Git

Использование Git для внесения существенных изменений, таких как:

* Способствуя коду.
* Внесение изменений, влияющих на значение.
* Внесение больших изменений в текст.
* Добавление новых тем.

### <a name="to-contribute-using-git"></a>Внести вклад с помощью Git

1. Если у вас нет учетной записи GitHub, установите ее в [GitHub.](https://github.com/join)
2. После того как у вас есть учетная запись, установите Git на компьютере. Выполните действия в [руководстве Настройка Git.]
3. Чтобы отправить запрос на тягу с помощью Git, выполните действия в [Use GitHub, Git и этом репозитории.](#use-github-git-and-this-repository)
4. Вам будет предложено подписать лицензионный договор участника, если вы:

    * Член группы Microsoft Open Technologies.
    * Вкладчик, который не работает в Корпорации Майкрософт.

В качестве члена сообщества необходимо подписать Соглашение о лицензии на вклад (CLA), прежде чем вы сможете внести большие материалы в проект. Необходимо выполнить и отправить документацию только один раз. Внимательно просмотрите документ. Может потребоваться подпись вашего работодателя.

Подписание cla не предоставляет вам права на фиксацию в основном репозитории, но это означает, что команды разработчика и разработчика office Content Publishing смогут просмотреть и утвердить ваши вклады. Вы засчет для ваших представлений.

Запросы на вытяги, как правило, рассматриваются в течение 10 бизнес-дней.

## <a name="use-github-git-and-this-repository"></a>Использование GitHub, Git и этого репозитория

**Примечание.** Большая часть сведений в этом разделе можно найти в статьях [GitHub Help.]  Если вы знакомы с Git и GitHub, перенаправить раздел Contribute и **изменить** содержимое для специфики потока кода и контента этого репозитория.

### <a name="to-set-up-your-fork-of-the-repository"></a>Настройка вилки репозитория

1. Чтобы добавлять информацию в этот проект, настройте учетную запись GitHub. Если вы этого не сделали, перейдите в [GitHub](https://github.com/join) и сделайте это сейчас.
2. Установка Git на компьютере. Выполните действия в [руководстве Настройка Git.]
3. Создайте свое ответвление этого репозитория. Для этого в верхней части страницы выберите кнопку **Fork.**
4. Скопируйте вилкой на компьютер. Для этого откройте Git Bash. Введите в командной строке следующую команду:

        git clone https://github.com/<your user name>/<repo name>.git

    Затем создайте ссылку на корневой репозиторий с помощью следующих команд:

        cd <repo name>
        git remote add upstream https://github.com/OfficeDev/<repo name>.git
        git fetch upstream

Поздравляем! Вы настроили свой репозиторий. Вам не потребуется повторять эти действия.

### <a name="contribute-and-edit-content"></a>Добавление и редактирование содержимого

Чтобы сделать процесс вклада максимально плавным, выполните следующие действия.

#### <a name="to-contribute-and-edit-content"></a>Для внося вклад и редактирования контента

1. Создайте новую ветвь.
2. Добавьте или отредактируйте содержимое.
3. Отправьте запрос на включение внесенных изменений в основной репозиторий.
4. Удалите ветвь.

**Важно** Ограничиваем каждую ветвь одной концепцией или статьей, чтобы оптимизировать рабочий поток и снизить вероятность конфликтов слияния. Содержимое, подходящее для новой ветви, включает в себя:

* Новая статья.
* Правки орфографии и грамматики.
* Применение одного изменения форматирования для большого набора статей (например, применения нового подножки авторских прав).

#### <a name="to-create-a-new-branch"></a>Создание новой ветви

1. Откройте Git Bash.
2. В командной подсказке Git Bash введите `git pull upstream master:<new branch name>` . Это создает локально новую ветвь, скопированную из последнего магистратного филиала OfficeDev.
3. В командной подсказке Git Bash введите `git push origin <new branch name>` . Это оповещает GitHub о новом филиале. Новая ветвь появится в вашем ответвлении репозитория на сайте GitHub.
4. В командной подсказке Git Bash введите переход `git checkout <new branch name>` на новую ветвь.

#### <a name="add-new-content-or-edit-existing-content"></a>Добавление или редактирование содержимого

Перейдите в репозиторий на компьютере с помощью проводника файлов. Файлы репозиториев находятся в `C:\Users\<yourusername>\<repo name>` .

Чтобы изменить файлы, откройте их в редакторе по вашему выбору и измените их. Чтобы создать новый файл, используйте редактор по вашему выбору и сохраните новый файл в соответствующем расположении в локальной копии репозитория. Во время работы часто сохраните работу.

Файлы являются рабочей копией новой ветви, созданной `C:\Users\<yourusername>\<repo name>` в локальном репозитории. Изменение файлов в этой папке не влияет на локальный репозиторий, пока вы не сохраните изменение. Чтобы сохранить изменение в локальном репозитории, введите следующие команды в GitBash:

    git add .
    git commit -v -a -m "<Describe the changes made in this commit>"

Команда `add` добавляет изменения в область промежуточного хранения для подготовки к сохранению в репозитории. Период после того, как команда указывает, что необходимо этапировать все файлы, которые вы добавили или изменили, повторно проверяя `add` подстановки. (Если вы не хотите сохранять все изменения, вы можете добавить определенные файлы. Вы также можете отменить изменения. Для получения справки введите команду `git add -help` или `git status`.)

Команда `commit` применяет промежуточные изменения в репозитории. Переключатель `-m` означает, что вы предоставляете коммит-комментарий в командной строке. Переключатели -v и -a могут быть опущены. Переключатель -v — для многословного вывода из команды, и -a делает то, что вы уже сделали с командой добавления.

Вы можете совершить несколько раз, пока вы делаете свою работу, или вы можете совершить один раз, когда вы сделали.

#### <a name="submit-a-pull-request-to-the-main-repository"></a>Отправка запроса на включение внесенных изменений в основной репозиторий

Когда вы закончите работу и будете готовы к ее слиянию в основной репозиторий, выполните следующие действия.

#### <a name="to-submit-a-pull-request-to-the-main-repository"></a>Отправка запроса на оттяги в основной репозиторий

1. В командной подсказке Git Bash введите `git push origin <new branch name>` . `origin` — это репозиторий GitHub, с которого вы скопировали свой локальный репозиторий. Эта команда передает текущее состояние новой ветви, в том числе все изменения, сохраненные на предыдущих шагах, в вашем ответвлении GitHub.
2. На сайте GitHub перейдите к новой ветви в своем ответвлении.
3. Выберите **кнопку Pull Request** в верхней части страницы.
4. Убедитесь, что ветвь Базы `OfficeDev/<repo name>@master` есть, а головной — `<your username>/<repo name>@<branch name>` .
5. Выберите **кнопку "Диапазон фиксации обновления".**
6. Добавьте название в запрос на тягу и опишите все внесенные изменения.
7. Отправьте запрос на включение внесенных изменений.

Один из администраторов сайта обработать запрос на вытягивать. Запрос на тягу будет всплыть на сайте OfficeDev/на <repo name> сайте Issues. Когда запрос на вытягу будет принят, проблема будет устранена.

#### <a name="create-a-new-branch-after-merge"></a>Создание ветви после объединения

После успешного слияния ветвей (т. е. ваш запрос на тягу принимается), не продолжайте работать в этом локальном филиале. Это может привести к слиянию конфликтов при отправке другого запроса на оттяги. Чтобы сделать еще одно обновление, создайте новую локализованную ветвь из успешно слитого вверх по течению филиала и удалите начальный локальный филиал.

Например, если локализованная ветвь X была успешно объединена в магистратуру OfficeDev/microsoft-graph-docs и необходимо внести дополнительные обновления в слитый контент. Создание нового локального филиала X2 из магистратуры OfficeDev/microsoft-graph-docs. Для этого откройте GitBash и выполните следующие команды:

    cd microsoft-graph-docs
    git pull upstream master:X2
    git push origin X2

Теперь у вас есть локальные копии (в новом локальном филиале) работы, отправленной в филиале X. В Ветвь X2 также содержит все работы, слитые другими авторами, поэтому если ваша работа зависит от работы других (например, общих образов), она доступна в новой ветви. Вы можете убедиться, что ваша предыдущая работа (и другие) находится в филиале, проверив новую ветвь...

    git checkout X2

…и проверив ее содержимое. (Команда `checkout` обновляет файлы до `C:\Users\<yourusername>\microsoft-graph-docs` текущего состояния ветви X2.) После проверки новой ветвь можно сделать обновления контента и совершить их как обычно. Однако во избежание работы в объединенной ветви (X) рекомендуем удалить ее (см. раздел **Удаление ветви** ниже).

#### <a name="delete-a-branch"></a>Удаление ветви

После успешного слияния изменений в основной репозиторий удалите используемую ветку, так как она больше не нужна.  Любая дополнительная работа должна быть проделана в новом филиале.  

#### <a name="to-delete-a-branch"></a>Удаление филиала

1. В командной подсказке Git Bash введите `git checkout master` . Это гарантирует, что вы не работаете в удаляемой ветви (это не допускается).
2. Далее в командной подсказке введите `git branch -d <branch name>` . Это удаляет ветвь на компьютере только в том случае, если она успешно слилась с репозиторием upstream. (Вы можете переопределить это правило с помощью флага `–D`, но для начала убедитесь, что это необходимо.)
3. Наконец, введите в командной строке команду `git push origin :<branch name>` (с пробелом перед двоеточием и без пробела после него).  При этом ветвь будет удалена из вашего ответвления GitHub.  

Поздравляем, вы успешно внесли свой вклад в проект!

## <a name="faq"></a>Вопросы и ответы

### <a name="how-do-i-get-a-github-account"></a>Как создать учетную запись GitHub?

Заполните форму на странице [Join GitHub](https://github.com/join), чтобы создать бесплатную учетную запись GitHub.

### <a name="where-do-i-get-a-contributors-license-agreement"></a>Где можно найти лицензионное соглашение с участником?

Вам будет автоматически отправлено уведомление, если для включения внесенных вами изменений вам нужно подписать лицензионное соглашение с участником (CLA).

**Вам нужно подписать лицензионное соглашение с участником (CLA), прежде чем вы сможете вносить большие изменения в этот проект**. Вам нужно заполнить и отправить документ всего один раз. Внимательно просмотрите документ. Может потребоваться подпись вашего работодателя.

### <a name="what-happens-with-my-contributions"></a>Что происходит с моими взносами?

При отправке изменений с помощью запроса на оттяги наша команда будет уведомлена и рассмотрит ваш запрос на вытяги. Вы будете получать уведомления о запросе на оттяги из GitHub; Вы также можете быть уведомлены кем-то из нашей команды, если нам потребуется больше информации. Если ваш запрос на вытягу будет утвержден, мы обновим документацию. Мы оставляем за собой право изменить отправку для юридических, стильных, четких или других проблем.

### <a name="can-i-become-an-approver-for-this-repositorys-github-pull-requests"></a>Могу ли я стать одобрением запросов на вытягивать GitHub этого репозитория?

В настоящее время внешние участники не могут утверждать запросы на оттяги в этом репозитории.

### <a name="how-soon-will-i-get-a-response-about-my-change-request"></a>Как скоро я получаю ответ о моем запросе на изменение?

Запросы на вытяги, как правило, рассматриваются в течение 10 бизнес-дней.
