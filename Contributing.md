## <a name="contribute-using-github"></a>Внесение дополнений с помощью GitHub

GitHub можно использовать для внесения дополнений в эту документацию без клонирования репозитория на компьютере. Это самый простой способ создать запрос на включение внесенных изменений в этот репозиторий. Используйте его для внесения небольших изменений, не затрагивающих код. 

**Примечание** Этот метод допускает единовременное внесение изменений в одну статью.

### <a name="to-contribute-using-github"></a>Внесение дополнений с помощью GitHub

1. Найдите статью, в которую следует внести дополнения, на сайте GitHub.
2. Открыв статью на сайте GitHub, выполните вход (вы можете бесплатно получить учетную запись, [присоединившись к GitHub](https://github.com/join)).
3. Нажмите на **значок карандаша** (редактирование файла в своем ответвлении этого проекта) и внесите свои изменения в окне **<>Edit file**. 
4. Прокрутите страницу вниз и введите описание.
5. Последовательно нажмите на кнопки **Предложить изменение файла**> **Создать запрос на включение внесенных изменений**.

Теперь вы успешно отправили запрос на включение внесенных изменений. Как правило, запросы на включение внесенных изменений проверяются в течение 10 рабочих дней. 


## <a name="contribute-using-git"></a>Внесение дополнений с помощью Git

С помощью Git можно призводить существенные изменения, например:

* Добавление кода.
* Внесение изменений, влияющих на смысл статьи.
* Внсеение значительных изменений в текст.
* Добавление новых статей.

### <a name="to-contribute-using-git"></a>Внесение дополнений с помощью Git

1. Если у вас нет учетной записи GitHub, создайте ее на сайте [GitHub](https://github.com/join). 
2. Создав учетную запись, установите Git на компьютере. Выполните действия, описанные в [руководстве по установке Git].
3. Чтобы отправить запрос на включение внесенных изменений с помощью Git, выполните действия, указанные в разделе [Использование GitHub, Git и этого репозитория](#use-github-git-and-this-repository).
4. Вам будет предложено подписать лицензионное соглашение для участников, если вы:

    * Являетесь членом группы Microsoft Open Technologies.
    * Являетесь участником, не работающим в корпорации Майкрософт.

Перед внесением значительных изменений в проект вам, как члену сообщества, необходимо подписать лицензионное соглашение для участников (CLA). Заполнить и отправить документ достаточно один раз. Внимательно просмотрите документ. Может потребоваться подпись вашего работодателя.

Подписывая лицензионное соглашение для участников (CLA), вы не получаете прав на внесение изменений в основной репозиторий, но члены группы разработчиков Office и группы публикации контента для разработчиков Office смогут просматривать и утверждать ваши изменения. Ваш вклад будет оценен по достоинству.

Как правило, запросы на включение внесенных изменений проверяются в течение 10 рабочих дней.

## <a name="use-github-git-and-this-repository"></a>Использование GitHub, Git и этого репозитория

**Примечание.** Большую часть сведений из этого раздела можно найти в статьях[Помощь по GitHub].  Если вы знакомы с Git и GitHub, переходите к разделу **Добавление и редактирование содержимого**, где описывается порядок раьоты с кодом и содержимым этого репозитория.

### <a name="to-set-up-your-fork-of-the-repository"></a>Настройка ответвления в репозитории

1.  Для внесения дополнений к этому проекту настройте учетную запись GitHub. Если вы еще не сделали этого, перейдите на сайт [GitHub](https://github.com/join) и создайте ее.
2.  Установите Git на своем компьютере. Выполните действия, описанные в [руководстве по установке Git].
3.  Создайте свое ответвление этого репозитория. Для этого нажмите кнопку **Вилка** в верхней части страницы.
4.  Скопируйте свое ответвление на компьютер. Для этого откройте Git Bash. Введите в командной строке следующую команду:

        git clone https://github.com/<your user name>/<repo name>.git

    Затем создайте ссылку на корневой репозиторий с помощью следующих команд:

        cd <repo name>
        git remote add upstream https://github.com/OfficeDev/<repo name>.git
        git fetch upstream

Поздравляем! Вы настроили свой репозиторий. Вам не потребуется повторять эти действия еще раз.

### <a name="contribute-and-edit-content"></a>Добавление и редактирование содержимого

Чтобы добавлять и редактировать содержимое было удобнее, выполните указанные ниже действия.

#### <a name="to-contribute-and-edit-content"></a>Добавление и редактирование содержимого

1. Создайте новую ветвь.
2. Добавьте новое или отредактируйте существующее содержимое.
3. Отправьте запрос на включение внесенных изменений в основной репозиторий.
4. Удалите ветвь.

**Важно!** Чтобы упростить рабочий процесс и снизить риск возникновения конфликтов при объединении, постарайтесь, чтобы каждая ветвь была связана с одним понятием или одной статьей. Для новой ветви может использоваться следующее содержимое:

* новая статья;
* Внесение изменений, связанных с орфографией и грамматикой.
* Применение единообразного форматирования для большого количества статей (например, применение нового нижнего колонтитула с информацией об авторских правах).

#### <a name="to-create-a-new-branch"></a>Создание новой ветви

1.  Откройте Git Bash.
2.  В командной строке Git Bash введите команду `git pull upstream master:<new branch name>`. При этом создается локальная ветвь, которая копируется из последней главной ветви OfficeDev.
3.  В командной строке Git Bash введите команду `git push origin <new branch name>`. Это позволит уведомить GitHub о наличии новой ветви. После этого новая ветвь появится в вашем ответвлении репозитория на сайте GitHub.
4.  Введите в командной строке Git Bash команду `git checkout <new branch name>`, чтобы переключиться на новую ветвь.

#### <a name="add-new-content-or-edit-existing-content"></a>Добавление или редактирование содержимого

Вы можете перейти к репозиторию на компьютере с помощью проводника. Файлы репозитория находятся в папке `C:\Users\<yourusername>\<repo name>`.

Чтобы отредактировать файлы, откройте их в любом редакторе и внесите требуемые изменения. Чтобы создать новый файл, воспользуйтесь любым редактором и сохраните новый файл в подходящем расположении, в локальной копии репозитория. Периодически сохраняйте файлы во время работы.

Файлы в папке `C:\Users\<yourusername>\<repo name>` представляют собой рабочую копию новой ветви, созданной в локальном репозитории. Модификация файлов, содержащихся в этой папке, не влияет на локальный репозиторий до сохранения изменений. Чтобы сохранить изменение в локальном репозитории, введите в GitBash следующие команды:

    git add .
    git commit -v -a -m "<Describe the changes made in this commit>"

Команда `add` добавляет внесенные изменения в область промежуточного хранения для подготовки к их сохранению в репозитории. Точка после команды `add` указывает на то, что на промежуточное хранение следует поместить все добавленные и измененные файлы, выполнив рекурсивную проверку вложенных папок. (Чтобы не сохранять все внесенные изменения, можно добавить определенные файлы. Также сохранение изменений можно отменить. Для получения справки введите команду `git add -help` или `git status`).

Команда `commit` позволяет внести данные, находящиеся на промежутоном хранении, в репозиторий. Параметр `-m` указывает, что в командной строке введен комментарий к изменению. Параметры -v и -a указывать необязательно. Параметр -v обеспечивает вывод подробных данных при выполнении команды, а параметр -a выполняет те же действия, что и команда add, запущенная ранее.

Вы можете сохранять изменения несколько раз во время работы или только один раз, когда все будет готово.

#### <a name="submit-a-pull-request-to-the-main-repository"></a>Отправка запроса на включение внесенных изменений в основной репозиторий

Кода вы завершите работу и подготовитесь к объединению внесенных изменений с главным репозиторием, выполните указанные ниже действия.

#### <a name="to-submit-a-pull-request-to-the-main-repository"></a>Как отправить запрос на включение внесенных изменений в основной репозиторий

1.  В командной строке Git Bash введите команду `git push origin <new branch name>`. Параметр `origin` в локальном репозитории будет указывать на репозиторий GitHub, их которого производилось клонирование локального репозитория. Эта команда передает текущее состояние новой ветви, включая все сохраненные ранее изменения, вашему ответвлению GitHub.
2.  На сайте GitHub перейдите к новой ветви в своем ответвлении.
3.  Нажмите на кнопку **Запрос на включение внесенных изменений** в верхней части страницы.
4.  Убедитесь в том, что в качестве базовой ветви указана ветвь `OfficeDev/<repo name>@master`, а в качестве ветви заголовка — `<your username>/<repo name>@<branch name>`.
5.  Нажмите на кнопку **Обновить диапазон сохраненных изменений**.
6.  Добавьте название запроса на включение внесенных изменений и опишите сделанные изменения.
7.  Отправьте запрос на включение внесенных изменений.

Его обработает один из администраторов сайта. Ваш запрос появится в разделе «Проблемы» на сайте OfficeDev/<repo name>. Когда запрос будет принят, проблема будет решена.

#### <a name="create-a-new-branch-after-merge"></a>Создание ветви после объединения

После успешного присоединения ветви (т. е. подтверждения запроса на включение внесенных изменений) работать с этой локальной ветвью не следует. Подача еще одного запроса может привести к конфликтам при объединении. Чтобы внести еще одно изменение, создайте новую локальную ветвь на основе ветви, успешно добавленной ранее, и удалите исходную локальную ветвь.

Так, к примеру, для внесения дополнительных изменений в объединенное содержимое после успешного объединения локальной ветви X с главной ветвью OfficeDev/microsoft-graph-docs необходимо создать локальную ветвь X2 на основе главной ветви OfficeDev/microsoft-graph-docs. Для этого следует открыть GitBash и выполнить следующие команды:

    cd microsoft-graph-docs
    git pull upstream master:X2
    git push origin X2

Теперь у вас есть локальные копии изменений (содержащиеся в новой локальной ветви), отправленных в ветви X. Кроме того, ветвь X2 содержит все изменения, внесенные другими авторами, таким образом, если в вашей работе используются чужие материалы (например, общие изображения), они будут доступны в новой ветви. Убедиться в наличии в новой ветви внесенных ранее изменений (и изменений, внесенных другими) можно, открыв эту ветвь...

    git checkout X2

...и проверив содержимое. (Команда `checkout` обновляет файлы в папке `C:\Users\<yourusername>\microsoft-graph-docs` до текущего состояния ветви X2). После проверки новой ветви можн вносить изменения в содержимое и выполнять их сохранение стандартным способом. Однако во избежание ошибочной модификации объединенной ветви (X), ее рекомендуется удалить (см. раздел **Удаление ветви** ниже).

#### <a name="delete-a-branch"></a>Удаление ветви

Успешно объединив свои изменения с главным репозиторием, удалите соответствующую ветвь, так как она больше не требуется.  Всю дальнейшую работу следует выполнять в новой ветви.  

#### <a name="to-delete-a-branch"></a>Как удалить ветвь

1.  В командной строке Git Bash введите команду `git checkout master`. Это гарантирует, что вы не работаете в удаляемой ветви (это не допускается).
2.  Затем введите в командной строке команду `git branch -d <branch name>`. Это позволит удалить ветвь с вашего компьютере только в случае ее успешного объединения с репозиторием верхнего уровня. (Изменить этот режим выполнения можно с помощью флага `–D`, но перед выплнением этой операции следует убедиться в необходимости подобных действий).
3.  В завершение, ведите в командной строке команду `git push origin :<branch name>` (с пробелом перед двоеточием и без пробела после него).  При этом ветвь будет удалена из вашего ответвления GitHub.  

Поздравляем! Вы успешно внесли свой вклад в проект.

## <a name="faq"></a>Вопросы и ответы

### <a name="how-do-i-get-a-github-account"></a>Как получить учетную запись GitHub?

Заполните форму на странице [Присоединиться к GitHub](https://github.com/join), чтобы создать бесплатную учетную запись GitHub. 

### <a name="where-do-i-get-a-contributors-license-agreement"></a>Где можно найти лицензионное соглашение для участников? 

Вам будет автоматически отправлено уведомление, если для включения внесенных вами изменений вам нужно подписать лицензионное соглашение с участником (CLA). 

Перед внесением значительных изменений в данный проект **вам, как члену сообщества, необходимо подписать лицензионное соглашение для участников (CLA)**. Вам нужно заполнить и отправить документ всего один раз. Внимательно просмотрите документ. Может потребоваться подпись вашего работодателя.

### <a name="what-happens-with-my-contributions"></a>Что происходит со внесенными мной изменениями?

Когда вы вносите изменения с помощью запроса на включение внесенных изменений, наша команда получает соответствующее уведомление и проверяет запрос. Вы будете получать уведомления о запросе на включение внесенных изменений от GitHub; также, вы вы можете получить уведомления от членов нашей команды, если нам потребуются дополнительные сведения. Если ваш запрос будет одобрен, мы обновим документацию. Мы оставляем за собой право на редактирование внесенных вами измененийс целью исправления правовых и стилистических ошибок, внесения ясности и устранения других недостатков.

### <a name="can-i-become-an-approver-for-this-repositorys-github-pull-requests"></a>Могу ли я стать утверждающим для запросов на включение внесенных изменений в репозиторий GitHub?

В настоящее время мы не разрешаем внешним участникам утверждать запросы на включение внесенных изменений в этот репозиторий.

### <a name="how-soon-will-i-get-a-response-about-my-change-request"></a>Насколько быстро я получу ответ на свой запрос на включение изменений?

Как правило, запросы на включение внесенных изменений проверяются в течение 10 рабочих дней.
